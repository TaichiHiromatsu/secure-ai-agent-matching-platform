# NEDO質問への回答案：普及に向けた課題とプラン

---

## 質問

> 普及に関する課題・プランがあればお聞かせください

---

## 回答案

普及に向けた課題とプランについて、**3つの観点**からお伝えします。

---

### 1. 競合に対する差別化

Google・Microsoft等の大手プラットフォーマーがA2Aのセキュリティ層を自前で構築する可能性は十分にあります。しかし、本プラットフォームには**構造的な差別化要素**があります。

まず、大手プラットフォーマーは自社エコシステム内のセキュリティに注力するため、**ベンダー横断の中立的な信頼評価**は提供しにくい立場にあります。GoogleがGoogle製エージェントに高スコアを付ける評価を、他社が信頼するでしょうか。本プラットフォームはOSSかつ特定ベンダーに依存しない設計であり、この**中立性こそが最大の強み**です。

加えて、**仲介エージェントの多層防御構造**（Matcher→Planner→Orchestrator→Anomaly Detector→Final Anomaly Detector）は、単なるフィルタリングではなく、計画と実行を分離したうえでリアルタイムに逸脱を検知する仕組みです。この設計は学術的裏付けのある手法を組み合わせたもので、容易に模倣しにくいものと考えています。

---

### 2. エージェント提供者と利用者をどう巻き込むか

普及における最大の課題は「鶏と卵の問題」です。ストアに登録エージェントが少なければ利用者は来ず、利用者がいなければ提供者も登録しません。この課題に対し、**提供側（toB）と利用側（toC）で異なるアプローチ**を考えています。

**【提供側：登録すること自体にメリットがある設計】**

エージェントストアへの登録プロセスは、AISI準拠の安全性評価を含む包括的なセキュリティ審査です。つまり**登録すること自体が、自社エージェントの品質証明**になります。提供者にとっては、審査結果をセキュリティ診断レポートとして活用でき、顧客への説明材料にもなります。信頼スコアのバッジをAgent Card上に表示可能にすれば、ストアに載っていること自体がブランド価値となり、登録のインセンティブが生まれます。

**【利用側：ユーザーエージェントへの組み込みが鍵】**

toC普及の鍵は、**ユーザーが本プラットフォームの存在を意識しなくても恩恵を受けられる構造**を作ることです。そのためには、エンドユーザーが日常的に使うAIアシスタント（＝ユーザーエージェント）が、裏側で仲介エージェントとエージェントストアを経由して外部エージェントを呼び出す形で組み込まれる必要があります。

ここで課題となるのが、**国内にA2A対応のtoCユーザーエージェントがまだ存在しない**という現実です。A2Aプロトコル自体がv0.3の黎明期であり、本番環境での相互運用実績はほぼありません。これは本プラットフォームだけの課題ではなく、A2Aエコシステム全体の「鶏と卵の問題」です。

この課題に対し、国内で現実的な連携先として**3つの層**を想定しています。

**① toC層：LINEヤフー**
Yahoo! JAPANアプリの「AIアシスタント」が既にお買い物・おでかけなどの領域でエンドユーザー向けに稼働しており、ソフトバンク・東京大学との「Beyond AI技術研究組合」でパーソナルAIエージェントの研究も進めています。LINE + Yahoo!というユーザー基盤は国内最大級であり、ここがA2A対応してストアを裏で利用する形になれば、一般消費者への普及の最短ルートになると考えています。

**② toB層：NEC・クラウドエース等**
NECはGoogle Cloudと協業しADK + A2Aでエージェント開発・提供を計画しています。クラウドエースも「Agent Enterprise」でA2A + MCPの開発に着手しました。エンタープライズ向けのAIエージェントがストア経由で外部エージェントを調達する形なら、toBでの導入は比較的早い段階で実現可能です。

**③ 行政層：デジタル庁との接点**
デジタル庁が開発中の行政専用AI環境「源内（GenAI）」は職員専用のため一般国民への展開はありませんが、行政内部で利用されるエージェントの安全性審査基盤としてストアが採用される可能性はあります。AISIとの協業の延長線上で、行政側への提案も視野に入れています。

いずれの層でも、エンドユーザーはストアの存在を意識する必要がなく、「なぜか安全に使える」という体験になります。**toBのプラットフォーマーおよび国内主要プレイヤーへのアプローチに集中する**のが、現時点で最も現実的な普及戦略です。

**【初期カタログの立ち上げ】**

利用側の連携先が見つかっても、ストアに登録エージェントがなければ意味がありません。この「最初の10エージェントを誰が登録するか」問題に対しては、GENIAC/NEDOの枠組みでパートナー企業に実証参加を呼びかけ、初期カタログを共同で構築するアプローチを想定しています。登録＝品質証明というインセンティブ設計と合わせて、供給側の初動を作ります。

### 3. 公的機関の後ろ盾の獲得と国際標準化活動

信頼スコアの権威性は、プラットフォームの普及を左右する最重要要素です。一スタートアップが付与するスコアだけでは、企業が安心して採用判断の根拠にはできません。

そのため、**国内での公的な後ろ盾の獲得**と**国際的な標準化活動への参画**の両面から進めたいと考えています。

**【国内：NEDO・AISIとの協業】**

**第1段階**として、AISIが進めるAIセーフティ評価の**公式テストベッド**としての認定を目指します。本プラットフォームのエージェントストア審査はAISIのAISEV 10観点評価を基に設計しており、AISIの基準を実装レベルで体現しています。

**第2段階**として、NEDO等の事業化支援プログラムを活用しながら、金融・医療・行政など**規制産業での実証実験**を行い、具体的な成功事例を蓄積します。

**第3段階**として、これらの実績を元に、国内の「エージェント信頼フレームワーク」の**標準仕様**策定への参画を目指します。国の基準として認められれば、スコアは「一企業の評価」から「国の基準に基づく評価」へと格上げされ、普及の大きな追い風になると考えています。

**【国際：AAIF（Agentic AI Foundation）への参画】**

A2A・MCP等のエージェント間標準を管轄するLinux Foundation傘下の**AAIF**への参画も視野に入れています。AAIFにはAnthropic・Google・Microsoft等がPlatinumメンバーとして参加し、Silver会員（年$10K）にはAirbyteやPydantic等のOSSプロジェクト主体の企業も名を連ねています。

AAIFに参画する意義は、Trusted Agent Storeの信頼評価の仕組みを**A2Aプロトコルの標準セキュリティレイヤーとして提案**できる点にあります。国内でAISIとの実績を作りつつ、国際的にはAAIFの標準化プロセスの中で自然にプラットフォームの名前が出る — この**国内実績と国際標準化の好循環**が、普及の大きな推進力になると考えています。

また、AISIがAAIFの標準策定プロセスにオブザーバー的に関与する流れができれば、AISI準拠の評価基準が国際標準に組み込まれる可能性もあり、日本発のエージェント安全性評価が世界標準になるシナリオも描けます。

## （まとめ）

| 課題 | 対応方針 |
|------|---------|
| 大手プラットフォーマーとの競合 | ベンダー中立・OSS・仲介エージェントの構造的差別化 |
| エージェント提供者の巻き込み | 登録＝品質証明という設計、信頼スコアバッジ |
| 初期カタログの不在 | GENIAC/NEDOの枠組みでパートナー企業と共同構築 |
| A2A対応ユーザーエージェントの不在 | LINEヤフー（toC）・NEC/クラウドエース（toB）・デジタル庁（行政）の3層へアプローチ |
| エンドユーザー（toC）への普及 | ユーザーエージェントへの裏側組み込みで、ユーザーは意識不要 |
| 信頼スコアの権威性 | NEDO・AISIとの協業→テストベッド認定→標準仕様化 |
| 国際標準化 | AAIF（Linux Foundation）への参画、A2Aセキュリティレイヤーとして提案 |

普及の核心は、**ユーザーに見えないインフラとして浸透すること**、**国内の主要プレイヤーとの連携でA2Aエコシステムを共に育てること**、**公的機関の後ろ盾で信頼の基盤を作ること**、そして**AAIFを通じた国際標準化で日本発の評価基準をグローバルに展開すること**です。GENIAC-PRIZEでの入賞は、これらを実現するための重要な第一歩と位置付けています。
