# 今後の展望：AgentCard CA署名によるエージェント真正性保証

## 背景と問題意識

現在の A2A エコシステムでは、エージェントが `/.well-known/agent-card.json` に自己申告で情報を公開する。これだけでは**「このエージェントは本当に安全か」を第三者が証明できない**という本質的な問題がある。

```
現状の問題：
  エージェント提供者 → "私は安全です" → AgentCard に自己申告
                                         ↓
  仲介エージェント → そのまま信じるしかない（野良エージェント排除不可）
```

A2A Protocol v0.3.0（2025/7/30）で `signatures[]` フィールドが追加され、
**第三者機関がAgentCardに署名できる仕組み**が仕様に組み込まれた。

---

## 提案：本プラットフォームが第三者CA として機能する

### 基本的な考え方

本プラットフォームはすでに多段階の安全性審査（Security Gate + MAGI陪審）を実施している。
この**審査結果を暗号的署名として刻む**ことで、評価の結果を機械検証可能な証拠に変える。

> 評価した → MAGI が approve した → **それを署名として証明する**

---

## アーキテクチャ：レジストリ保持方式（Option A）

### なぜレジストリ保持方式か

AgentCard はエージェント提供者のサーバーが配信するため、
プラットフォームが直接書き込むことはできない。

そこで、**プラットフォームが署名済みコピーを自身のレジストリに保管し、
仲介エージェントはそのレジストリを正典（Source of Truth）として使う**設計とする。

### フロー全体図

```
【登録・審査フェーズ】

エージェント提供者
  │
  │  AgentCard を提出（登録申請）
  ▼
Platform Registry
  │
  ├─ Security Gate（攻撃プロンプト耐性テスト）
  ├─ Agent Card Accuracy（機能整合性テスト）
  └─ MAGI 陪審員評価（AISEV + SAIF 2.0 準拠）
       ├─ MELCHIOR (GPT-4o)   : ポリシー遵守性
       ├─ BALTHASAR (Claude)  : 安全性・漏洩リスク
       └─ CASPER (Gemini)     : ユーザ意図との整合性
            ↓ approve
  Platform CA 秘密鍵で AgentCard に署名
  署名済みコピーをレジストリに保存
       ↓
  ✅ 「MAGI 審査済み」AgentCard として登録完了

---

【仲介フェーズ】

エージェントAからの接続要求
  │
  ▼
仲介エージェント
  │  Platform Registry から署名済み AgentCard を取得
  │  Platform の公開鍵（JWKS）で signatures[] を検証
  │
  ├─ ✅ 署名有効 → エージェントBへ接続許可
  └─ ❌ 署名なし / 署名不正 → 接続拒否（野良エージェント排除）
```

---

## 技術的な詳細

### AgentCard の signatures[] フィールド（A2A v0.3.0+）

```json
{
  "name": "example-agent",
  "url": "https://agent.example.com/a2a",
  "skills": [...],
  "signatures": [
    {
      "protected": "eyJhbGciOiJFUzI1NiIsInR5cCI6IkpPU0UiLCJqa3UiOiJodHRwczovL3BsYXRmb3JtLmV4YW1wbGUvandrcyJ9",
      "signature": "MEUCIQDx...",
      "header": {
        "x-platform": "secure-ai-agent-matching-platform",
        "x-magi-verdict": "approve",
        "x-evaluated-at": "2026-03-02T00:00:00Z"
      }
    }
  ]
}
```

**保護ヘッダー（Base64urlデコード後）：**

```json
{
  "alg": "ES256",
  "typ": "JOSE",
  "jku": "https://platform.example/jwks",
  "kid": "platform-ca-2026"
}
```

- `jku` : プラットフォームの公開鍵セット（JWKS）URL → 誰でも署名検証可能
- `signatures[]` は配列なので、提供者署名と本プラットフォームの署名が共存できる

### 提出後の改ざん検知

レジストリに保存する際、審査時点の AgentCard 全体をハッシュ化して署名対象とする。
これにより、**審査後に提供者が内容を変更しても検知できる**。

```
署名対象 = Hash(AgentCard JSON)
署名値   = ES256(Platform_CA_Private_Key, 署名対象)
```

## 期待される効果

| 課題 | 現状 | CA署名導入後 |
|------|------|-------------|
| 野良エージェントの排除 | 困難（自己申告のみ） | 署名なし = 仲介拒否で完全排除 |
| エージェントの真正性証明 | なし | MAGI審査済みを暗号的に証明 |
| なりすましリスク | 高い | 署名検証で即座に検知 |
| 審査結果の外部公開 | 口頭・UI表示のみ | 機械検証可能な形で配布 |
| A2A標準準拠 | 部分的 | signatures[] フィールド完全活用 |

## 本プラットフォームの立ち位置

A2A プロトコルが定義する「署名者の4層構造」において、本プラットフォームは
**③ エージェントレジストリ**の役割を担う。

```
① エージェント開発者 / 提供者   （自己署名）
② デプロイプラットフォーム      （例: Google Agent Engine）
③ エージェントレジストリ        ← 本プラットフォームが担う
④ 第三者監査機関 / 業界CA      （将来的な発展先）
```

現在、③の役割を果たすプレイヤーは世界的にも存在しない。
**AISEV + SAIF 2.0 に基づく AI 特化の安全性審査を行うレジストリ**として、
業界標準の CA へと発展する足がかりとなる。

## バージョン更新と再審査フロー

### 「バージョンが変わったら署名は無効では？」

**署名はハッシュベースなので、バージョン更新 = 署名無効になる。これは正しい動作である。**

AgentCard の内容が変われば署名対象のハッシュが変わり、プラットフォームの署名は検証エラーになる。
これにより、**エージェント提供者がレジストリの審査をバイパスしてコンテンツを書き換えることを防止**できる。

```
審査時   : AgentCard_v1 → MAGI approve → Hash(AgentCard_v1) に署名
バージョン更新後 : AgentCard_v2 → Hash 不一致 → 署名検証エラー → 自動的に「未審査」扱い
```

---

### 検証タイミング

**署名検証はセッション確立時の1回のみ行う。** メッセージ交換のたびに検証する必要はない。

これは TLS と同じ設計思想である：TLS ハンドシェイク時に証明書を検証し、以降の通信はその検証済みセッション内で行う。AgentCard の署名も同様に「このエージェントは信頼できるか」をセッション開始時に確認すれば十分であり、個々のメッセージの内容を保証するものではない。

また、レジストリへの問い合わせ結果は短時間キャッシュすることで、新規接続のたびにレジストリを叩くオーバーヘッドも軽減できる。

```
新規接続要求 → キャッシュを確認
  ├─ キャッシュ有効（TTL内、例: 5〜15分） → そのまま接続許可
  └─ キャッシュ切れ / 初回 → レジストリに問い合わせて署名検証 → 接続許可 / 拒否
```

---

### 変更検知フロー（仲介時）

キャッシュ切れ・初回接続のタイミングで、仲介エージェントは以下の手順で変更を検知する。

```
【仲介時の変更検知】

エージェントAからエージェントBへの接続要求
  │
  ▼
仲介エージェント
  │
  ├─ ① エージェントBの /.well-known/agent-card.json を取得（ライブ）
  │
  ├─ ② Registry の署名済みコピーと比較
  │    ├─ 第一チェック: version フィールドが異なる → 即座に変更検知
  │    └─ 第二チェック: Hash(live AgentCard) ≠ Hash(registry copy) → 変更検知
  │
  ├─ ✅ 一致 → 署名検証 → 接続許可（結果をキャッシュ）
  │
  └─ ❌ 不一致 → 変更検知 → 再審査ワークフロー起動
```

---

### 再審査ワークフロー

変更が検知された場合、以下のフローで再審査を実施する。

```
【再審査フロー】

変更検知
  │
  ▼
エージェントBの署名を「期限切れ」マーク（Registry 上）
  │
  ▼
再審査キューに追加
  │
  ├─ Security Gate（攻撃プロンプト耐性）
  ├─ Agent Card Accuracy（機能整合性）
  └─ MAGI 陪審員評価
       ├─ approve → 新しい AgentCard_v2 に署名 → Registry 更新 → 接続再開
       └─ reject  → Registry から削除 → 以降の接続を永続的に拒否
```

---

### 再審査中の接続状態ポリシー

| 状態 | 接続許可 | 説明 |
|------|----------|------|
| 初回登録前（未審査） | ❌ 不可 | 署名なし = 野良エージェント扱い |
| 審査済み・バージョン一致 | ✅ 可 | 正常状態 |
| バージョン不一致・再審査中 | ⚠️ 制限付き | 旧バージョンで approve 済みの実績があるため、再審査完了まで一時的に旧バージョンの署名で接続を許可する（運用ポリシーによる） |
| 再審査 reject | ❌ 不可 | 即座に接続拒否・Registry から削除 |

> **補足**: 「再審査中の接続をどうするか」は運用設計の重要なポイントである。
> 厳格なポリシーでは再審査完了まで全接続を停止し、寛容なポリシーでは旧バージョン承認を根拠に一時的な接続を維持する。
> 本プラットフォームでは後者（旧バージョン実績を信頼）を初期方針とし、高リスクカテゴリのエージェントには厳格ポリシーを適用することを検討する。
